# 第 3 部分 · 进阶使用技巧

这一章只讲普通用户最常用的几件事：提示词、上下文、图片、搜索/RAG，以及使用 AI 时的风险边界。先不用追求复杂技巧，记住一个核心：**AI 的表现取决于你给了什么信息，也取决于你限制它怎么使用这些信息。**

## 3.1 system prompt 和 user prompt

在网页聊天框里，你通常只看到一个输入框。但在 API 和很多专业工具里，一次请求会被拆成多条消息，最常见的是 `system` 和 `user`。

`system prompt` 是工作规则，像岗位说明：

```text
你是一个面向零基础读者的中文编辑。
请优先保证准确、清楚、简洁。
如果材料中没有依据，请标注“材料未提供”，不要补写事实。
```

`user prompt` 是当前任务：

```text
请把下面这段通知改写成适合发到微信群的版本。
要求：保留时间、地点和报名方式，不超过 180 字。
```

参考 [prompt-optimizer](https://github.com/linshenkx/prompt-optimizer) 这类工具，API 里常见的消息结构大概是：

```json
[
  { "role": "system", "content": "你是一个提示词优化助手。" },
  { "role": "user", "content": "请优化这段提示词：帮我写得好一点。" }
]
```

对普通用户来说，最重要的习惯是：**规则和任务分开写**。当你让 AI 修改一段提示词时，也要明确告诉它“这是待分析文本，不是要执行的任务”。

一个够用的模板：

```text
请帮我完成【任务】。

背景：
- 读者/使用场景：
- 我希望达到的效果：

材料：
……

要求：
1. 必须保留：
2. 不要：
3. 不确定的地方请标注：

输出格式：
……
```

## 3.2 写作、整理、翻译：重点是上下文控制

写作、整理、翻译看起来是三类任务，本质上都在考验你能不能给 AI 正确的上下文。

上下文包括你输入的话、前面几轮对话、上传的文件、截图、网页搜索结果和系统规则。AI 只能基于它看见的内容回答。不给材料，它就会猜；材料太杂，它会抓错重点。

写作时，不要一上来就让 AI “写一篇文章”。更稳的流程是：

1. 先让 AI 整理观点。
2. 再让 AI 生成提纲。
3. 你修改提纲，补充真实材料。
4. 最后让 AI 写正文或改稿。

例如：

```text
我想写一篇给零基础朋友看的文章，主题是“普通人怎么开始用 AI”。

我的观点：
- 不要一开始纠结模型排名，先学会提问。
- 日常使用从写作、整理、翻译、读图开始。
- 隐私和幻觉比技巧更重要。

请先不要写正文。请帮我整理成一个 5 段式提纲。
```

整理资料时，先定义结构：

```text
请整理下面的会议记录。

输出格式：
1. 一句话结论
2. 已确定事项
3. 待确认事项
4. 行动项表格：事项 / 负责人 / 截止时间

如果原文没有负责人或截止时间，请写“未提及”，不要猜。
```

翻译时，先说明用途：

```text
请把下面这段中文翻译成英文，用于发给海外合作方。
语气礼貌、直接，不要太营销。
专有名词 Codex、mdBook、GitHub Pages 保留原英文。
```

长对话还会遇到“遗忘”。它不是按轮数忘，而是按上下文长度忘。粗略估算：

| 每轮新增内容 | 典型场景 | 最早内容大概何时被挤出 |
| --- | --- | --- |
| 300 token | 日常短问短答 | 数百轮 |
| 1,200 token | 每轮贴一小段材料 | 几十到一百轮 |
| 3,000 token | 长文改稿、会议纪要 | 几十轮 |
| 10,000 token | 每轮上传长材料 | 十几轮内 |

真实产品还可能自动摘要或裁剪历史。长任务更稳的做法是定期生成“上下文快照”，再开新对话继续：

```text
请把目前对话压缩成一份继续工作的上下文快照：
1. 目标和读者
2. 已确定的事实
3. 风格规则
4. 待办事项
5. 仍不确定的问题
```

## 3.3 生成图片和读取图片

“让 AI 看图片”和“让 AI 生成图片”是两种能力。

读图是理解输入。AI 会把图片切块、编码，转换成模型能处理的视觉特征，再结合你的问题生成文字回答。它适合看截图、解释图表、描述画面、对比界面差异；但不适合完全依赖它做医学诊断、法律判断、身份识别或精确数小物体。

读图时可以这样问：

```text
请分析这张截图。
先只描述你能直接看到的事实。
如果有看不清的文字，请标注“看不清”，不要猜。
最后给我 3 个下一步操作。
```

生成图片是创造输出。产品背后通常不是聊天模型直接“画图”，而是一个流水线：语言模型理解需求，系统改写图片提示词，图像模型生成图片，再经过安全审核和后处理。所以一个产品能读图，不代表它一定能画图；能画图，也不代表画图用的是同一个模型。

图片提示词可以按这个结构写：

```text
主体：画面里最重要的对象
场景：在哪里，发生什么
构图：近景/远景、正面/俯视、主体位置
风格：照片、插画、手绘草图、产品渲染
色彩：主色和点缀色
限制：不要文字、不要水印、不要真实品牌 logo
用途：封面、PPT 插图、头像、教程配图
```

这里第一次出现“多模态 AI”的概念：文字、图片、音频、视频都是不同模态。能同时处理多种信息类型的 AI，就叫多模态 AI。

## 3.4 搜索和 RAG：本质是在编辑 prompt

很多 AI 产品看起来“知道最新信息”或“读懂了资料库”，本质上通常不是模型突然记住了一切，而是系统在回答前帮它找材料，并把材料放进上下文。

联网搜索的流程大概是：

1. 把你的问题改写成搜索关键词。
2. 搜索网页、新闻或文档。
3. 抽取相关片段。
4. 把片段放进 prompt。
5. AI 基于这些材料回答。

RAG 是 Retrieval-Augmented Generation，通常翻译为“检索增强生成”。它和搜索类似，只是检索范围通常是你的私有资料库，比如公司文档、课程讲义、PDF、客服记录、个人笔记。

一个简化后的 RAG prompt 可能长这样：

```text
system：
你是资料问答助手。只能基于给定材料回答。
材料没有的信息，请说“资料未提供”。

user：
问题：如何申请退款？

retrieved context：
[片段 1] 退款需在购买后 14 天内提交……
[片段 2] 企业订单需联系销售代表……
```

所以 RAG 的本质仍然是 prompt 编辑：系统自动把“可能相关的材料”加进上下文。

使用搜索或 RAG 时，不要只问答案，还要问依据：

```text
请基于资料回答。
每条结论后标注依据片段编号或来源链接。
如果资料不足，请写“资料不足”，不要推断。
```

## 3.5 常见问题：幻觉、隐私和边界

AI 最常见的问题不是“完全不能用”，而是“看起来很可信，但关键处可能错”。普通用户最需要注意三件事。

第一，抑制幻觉。重要事实不要让 AI 凭记忆回答，要让它区分依据和推测：

```text
请回答下面的问题。
要求：
1. 区分“有依据的结论”和“你的推测”。
2. 重要事实必须给出来源或引用材料编号。
3. 如果信息不足，请写“无法判断”，不要补全。
```

第二，控制隐私。不要随便上传身份证、护照、银行卡、病历、合同、工资单、客户名单、API Key、密码、验证码、私钥、未公开代码、聊天记录、人脸、车牌、地址和定位截图。上传前先裁剪、打码、改名、删字段。

尤其不要上传包含这些字段的配置文件：

```text
api_key
secret
token
password
private_key
DATABASE_URL
OPENAI_API_KEY
```

第三，识别高风险任务。医疗、法律、投资、税务、签证、考试政策、合同签署、对外公告，都不能直接把 AI 输出当最终答案。更稳的流程是：

```text
AI 起草 -> AI 自查风险 -> 人工核验关键事实 -> 必要时找专业人士确认 -> 再发布或执行
```

上传前用 10 秒检查：

1. 有没有身份信息、账号、地址、电话、人脸？
2. 有没有密码、密钥、二维码、验证码？
3. 有没有公司或他人的未公开信息？
4. 能不能只上传一小段样例？
5. AI 的答案会不会影响钱、健康、法律或工作机会？
