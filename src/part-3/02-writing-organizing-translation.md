# 写作、整理、翻译实战

写作、整理、翻译是普通用户最容易立刻用上 AI 的三类任务。它们看起来不同，本质上都在考验同一件事：你能不能控制好上下文。

这里的“上下文”不是一个抽象概念，而是 AI 当前能看到的全部信息：你刚输入的话、前面几轮对话、上传的文件、截图、网页搜索结果、系统规则，以及产品背后自动加入的说明。AI 的回答只能基于它看见的内容生成。如果你没有提供材料，它就会猜；如果你提供太多杂乱材料，它就会抓错重点。

## 写作：不要让 AI 一步到位

最常见的失败方式是直接说：

```text
帮我写一篇公众号文章，主题是 AI 学习。
```

它当然能写，但大概率会写成一篇通用、空泛、像模板的文章。更稳的做法是把写作拆成四步：

1. 让 AI 先帮你整理观点。
2. 让 AI 生成提纲。
3. 你修改提纲，补充真实材料。
4. 再让 AI 写正文或改稿。

可以这样开始：

```text
我想写一篇给零基础朋友看的文章，主题是“2026 年普通人怎么开始用 AI”。

我的观点：
- 不要一开始就纠结模型排名，先学会提问。
- 日常使用从写作、整理、翻译、读图开始。
- 隐私和幻觉比技巧更重要。

请先不要写正文。请帮我整理成一个 5 段式提纲，每段说明中心意思和需要补充的例子。
```

注意“请先不要写正文”。这句话很重要。它把 AI 从“自由发挥写文章”拉回到“帮我搭结构”。

## 改稿：告诉它改什么，不要只说“润色”

“润色”这个词太宽。AI 不知道你要它改语气、改逻辑、改错别字，还是压缩篇幅。你可以把改稿任务拆得更明确。

```text
请帮我改下面这段文字，只做三件事：
1. 删除重复表达。
2. 把长句拆短。
3. 保留原有事实，不新增例子。

请先列出你改动的原则，再给出改写版本。

原文：
……
```

如果你担心 AI 把你的意思改歪，可以要求它输出对照表：

```text
请用三列表格输出：原句、修改后、修改原因。
不要合并段落。
```

## 整理：先定义你要的结构

AI 很适合整理会议纪要、聊天记录、课程笔记、文章摘要。但你要先告诉它“整理成什么”。

例如整理会议记录：

```text
请整理下面的会议记录。

输出格式：
1. 一句话结论
2. 已确定事项
3. 待确认事项
4. 行动项表格：事项 / 负责人 / 截止时间 / 依据原文

规则：
- 不要补充原文没有的信息。
- 如果负责人或截止时间没有出现，请写“未提及”。
- 行动项必须引用原文中的一句依据。

会议记录：
……
```

这个提示词比“帮我总结一下”更可靠，因为它把 AI 的自由度限制在你需要的结构里。

整理资料时还有一个好习惯：让 AI 先做分类，再做总结。分类能暴露材料结构，总结才不容易混在一起。

```text
请先把下面的笔记按主题分类，不要总结。
每条笔记只能放入一个主题。如果你不确定，放到“待判断”。
```

## 翻译：先说明用途，再要求风格

翻译不是把每个词换成另一种语言。商务邮件、论文摘要、产品说明、字幕、社交媒体文案，对同一句话的翻译都不一样。

不够好的写法：

```text
翻译成英文。
```

更好的写法：

```text
请把下面这段中文翻译成英文，用于发给海外合作方。

要求：
1. 语气礼貌、直接，不要太营销。
2. 专有名词保留原英文：Codex、mdBook、GitHub Pages。
3. 如果中文里有含糊表达，请不要自行补事实，在译文后列出“需要确认的问题”。

原文：
……
```

如果是重要材料，建议加一个“回译检查”步骤：

```text
请先给英文译文，然后把英文直译回中文。
最后列出你认为可能改变原意的地方。
```

## 上下文会在多少轮之后被忘掉？

很多人以为 AI 聊久了会“记性变差”。更准确地说，有两种情况：

1. **软遗忘**：信息还在上下文里，但位置太靠前、太靠中间或被大量新内容淹没，模型没有稳定利用它。
2. **硬遗忘**：对话太长，最早的消息已经超出上下文窗口，产品不再把它发送给模型。

所以不存在一个固定答案，比如“第 20 轮一定忘”或“第 100 轮一定忘”。轮数取决于每轮有多长。

下面是一个可复现的 token 账本实验。假设一个聊天模型有 128k token 上下文窗口，我们预留 8k token 给回答，系统规则、工具说明和隐藏提示占用约 4k token，那么真正留给历史对话的空间大约是 116k token。

| 每轮新增内容规模 | 大概是什么场景 | 理论上第几轮开始挤出最早内容 |
| --- | --- | --- |
| 300 token / 轮 | 日常短问短答 | 约 380 轮后 |
| 1,200 token / 轮 | 每轮贴一小段材料并让它改 | 约 95 轮后 |
| 3,000 token / 轮 | 长文改稿、会议纪要、多段翻译 | 约 38 轮后 |
| 10,000 token / 轮 | 每轮都上传长材料或大段日志 | 约 11 轮后 |

这个表不是说第 38 轮之前一定不会忘，也不是说第 39 轮一定忘。它说明：对话寿命不是按轮数算，而是按 token 算。产品还可能提前做自动摘要、裁剪附件、只保留最近消息，所以实际表现会更复杂。

课堂上可以做一个简单测试：

```text
第 1 轮：请记住暗号“蓝色纸夹”，以后我问你暗号时必须原样回答。
第 2 到第 N 轮：每轮粘贴一段约 1,000 token 的会议材料，让 AI 总结。
每隔 10 轮：问“第一轮暗号是什么？第一轮还要求你遵守什么？”
```

当它开始只记得大意、答错暗号、或把后面材料里的词混进暗号时，就说明早期信息已经不稳定了。这个实验也能帮助你养成一个习惯：长任务不要无限续聊，要定期重建上下文。

## 长任务的上下文管理方法

持续写作或整理资料时，可以每隔一段时间让 AI 生成一份“上下文快照”：

```text
请把目前对话压缩成一份继续工作的上下文快照。

必须包含：
1. 已确定的目标和读者
2. 已采用的术语和风格规则
3. 不能改变的事实
4. 待办事项
5. 仍不确定的问题

请不要写成文章，只写成后续可直接粘贴给 AI 的工作说明。
```

然后你可以开一个新对话，把这份快照贴进去，再继续工作。这样比在一个超长对话里反复追问更稳。

## 本节练习

选择一个你真实的写作或整理任务，先不要让 AI 直接产出成品。按下面顺序做：

1. 让 AI 只整理提纲。
2. 你补充缺失背景。
3. 让 AI 输出第一版。
4. 让 AI 根据明确标准改稿。
5. 最后让 AI 生成一份上下文快照。

如果你发现第 5 步的快照已经偏离事实，说明前面的上下文控制还不够清楚。
